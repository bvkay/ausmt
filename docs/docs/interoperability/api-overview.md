# API Overview

> **Implementation status (current).** The REST API described on this page is a **planned
> interface** and is **not yet implemented**. AusMT is currently served as static files; the
> machine-readable discovery interface that exists *today* is the **MTCAT document** — a single
> JSON file (`mtcat.json`, conforming to `mtcat.schema.json`) generated by `ausmt`
> alongside the portal's `data/*.json`. Treat the resources and endpoints below as the target
> design for a future service, not as live URLs.

## Overview

The AusMT API provides machine-readable access to published metadata, survey packages and scientific products.

Its purpose is to support interoperability, discovery and integration with external software systems without requiring direct access to the underlying repositories.

The API complements the AusMT web portal by providing structured access to information that can be consumed by:

- Software applications
- Research workflows
- Data portals
- Visualisation tools
- Scientific processing systems

The API is intended to expose published information, not replace the survey package itself.

---

## Why an API?

Most users interact with AusMT through the web portal.

However, many scientific workflows require automated access to information.

Examples include:

- Building survey catalogues
- Synchronising metadata
- Generating maps
- Discovering datasets
- Integrating with external repositories
- Building custom applications

The API provides a consistent mechanism for accessing published information programmatically.

---

## API Principles

Several principles guide the design of the AusMT API.

### Read-Only Access

The API is primarily intended for discovery and access.

Published survey packages remain the authoritative record.

Submission, review and publication workflows occur through dedicated curation processes rather than through the public API.

---

### Published Products Only

The API exposes published information.

Examples include:

- Collections
- Survey metadata
- Station metadata
- Product metadata
- Provenance summaries
- Citation information

Unpublished or in-review material is not exposed.

---

### Stable Identifiers

API resources should be linked through persistent identifiers wherever possible.

Examples include:

- DOI
- ORCID
- ROR
- RAiD
- Survey identifiers
- Station identifiers

Stable identifiers improve interoperability between systems.

---

### Format Independence

The API should not require a particular transfer-function format.

A survey may contain:

- EDI
- EMTFXML
- MTH5

or multiple representations of the same transfer function.

The API describes the products that exist rather than prescribing which formats must be used.

---

## Information Model

The API reflects the AusMT information model:

```text
Collection
↓
Survey Package
↓
Station
↓
Transfer Function
↓
Derived Product
```

This hierarchy provides a consistent structure for discovery and navigation.

---

## Collections

Collection resources describe groups of related surveys.

Examples include:

- AusLAMP
- WAMT
- Institutional holdings
- State-based releases

Collection information may include:

- Title
- Description
- Geographic extent
- Custodian organisation
- Survey count

---

## Surveys

Survey resources are the primary API object.

Typical information includes:

- Survey identifier
- Title
- Abstract
- Geographic extent
- Acquisition dates
- Organisations
- Product availability
- Citation information

Most discovery workflows operate at the survey level.

---

## Stations

Station resources describe individual observation locations.

Examples include:

- Station identifier
- Coordinates
- Elevation
- Deployment dates
- Product availability

Station resources provide additional detail for users requiring site-level information.

---

## Products

Product resources describe the scientific products associated with a survey.

Examples include:

- EDI files
- EMTFXML files
- MTH5 products
- Quicklook products
- Phase tensor products
- Strike products
- Dimensionality products

The API allows users to determine what products are available before downloading them.

---

## Provenance

The API may expose provenance summaries describing:

- Product lineage
- Processing information
- Survey package versions
- Publication history

The objective is to support transparency and reproducibility while maintaining a lightweight discovery interface.

---

## Versioning

The API should expose survey package version information.

Examples include:

- Current version
- Publication date
- Update history

Users should be able to determine which version of a survey package they are accessing.

---

## Relationship to MTCAT

MTCAT and the API serve complementary purposes.

```text
MTCAT
↓
Metadata exchange

API
↓
Programmatic access
```

MTCAT provides a portable discovery model.

The API provides a mechanism for querying and retrieving published information.

Both are built upon the same underlying information model.

---

## Relationship to the Portal

The web portal and API expose the same published information.

The difference lies in presentation.

```text
Portal
↓
Human-readable

API
↓
Machine-readable
```

The portal and API should remain consistent and derive their information from the same published survey packages.

---

## Relationship to External Systems

The API is intended to support integration with external systems.

Examples may include:

- National data portals
- Research infrastructure platforms
- Institutional repositories
- Scientific software
- Discovery catalogues

The objective is to reduce barriers to reuse and integration.

---

## Authentication

Most discovery and metadata resources are expected to be publicly accessible.

Future implementations may include authenticated access for:

- Restricted datasets
- Embargoed products
- Administrative functions

Authentication requirements will depend on repository governance and access policies.

---

## Future Directions

The API is expected to evolve as AusMT develops.

New resource types, metadata fields and product categories may be added over time.

The underlying principles should remain stable:

- Published information
- Stable identifiers
- Consistent metadata
- Interoperability
- Programmatic access

---

## Principle

The purpose of the AusMT API is not to expose repository internals.

Its purpose is to provide a stable, machine-readable interface to published MT survey products and metadata.

Users should be able to discover, understand and access AusMT holdings programmatically without needing to know how the underlying repositories are organised or implemented.
