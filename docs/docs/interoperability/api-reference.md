# API Reference

> **Implementation status (current).** This page specifies a **planned** REST API. None of the
> `/v1/...` endpoints below are served yet. The interface that exists *today* is the static
> **MTCAT document** (`mtcat.json`, schema `mtcat.schema.json`) plus the portal's generated
> `data/*.json`. See [Current static interface](#current-static-interface).

## Overview

The AusMT API is a metadata and discovery service.

Its purpose is to expose information about collections, survey packages, stations and transfer-function availability.

The API is not intended to expose every derived product available through the portal.

## Information Model

```text
Collection
    ↓
Survey Package
    ↓
Station
    ↓
Transfer Function
```

## Core Resources (planned)

```text
/v1/collections
/v1/surveys
/v1/stations
```

Survey packages are the primary API resource. **These endpoints are not implemented yet** — use
the static MTCAT document below in the meantime.

## Current static interface

Until the REST service exists, the same information is available as static files generated by
the `engine` and published with the portal:

```text
data/mtcat.json        # MTCAT discovery document: portal, collections, surveys, stations
                       # (validates against schema/mtcat.schema.json)
data/surveys.json      # per-survey metadata
data/catalogue.json    # station catalogue (compact array form the portal renders)
```

`mtcat.json` is the recommended integration point: it is small, schema-versioned, and designed to
be harvested by other catalogues. The `/v1` resources above mirror its structure and are the
intended future replacement.

`catalogue.json`, `tf.json` and `sci.json` are **compact positional arrays** (read by index, not by
key). Every column is enumerated in the developer contract — see
[Portal Data Files](../developer/data-files.md) — which is the authoritative definition for anyone
adding or changing a field.

## Downloads

The API may expose resource links for published survey-package releases.

Custom station selection and polygon downloads are handled by the portal rather than the API.

## Principle

The API exposes metadata and resource resolution. The portal provides visualisation and interactive download workflows.
