# Architecture Overview

## Introduction

AusMT is organised as a collection of independent but connected components.

Each component has a clearly defined responsibility.

This separation is deliberate.

It allows scientific processing, data stewardship, publication and documentation to evolve independently while maintaining reproducibility and clear provenance.

At the highest level, AusMT is the `ausmt` monorepo (with `engine/`, `portal/`, `gateway/`,
`docs/`, `contract/`, `deploy/` and `maintainer/` subdirectories) plus the separate
`ausmt-surveys` data repository:

```text
submissions -> gateway/ -> ausmt-surveys -> engine/ -> portal/
```

Each component serves a different purpose.

---

## Design Principles

Several principles guide the architecture.

### Separation of Responsibilities

Scientific processing, data publication and user interfaces are separated.

The public portal should not perform scientific processing.

Scientific processing should not modify published products directly.

Documentation should not depend on implementation details.

---

### Reproducibility

Scientific products should be generated from documented workflows.

Where possible, derived products should be reproducible from the published survey package and processing software.

---

### Provenance

The origin of a product should be traceable.

Users should be able to determine:

- what was published
- where it came from
- how it was generated
- which version was used

---

### Interoperability

AusMT adopts existing community standards wherever practical.

Examples include:

- EDI
- EMTFXML
- MTH5
- mt_metadata

The architecture is designed to integrate with external repositories rather than replace them.

---

## Component Overview

### ausmt-surveys

The survey repository is the scientific backbone of AusMT.

It contains curated survey packages and associated scientific products.

Typical contents include:

- Survey metadata
- Station metadata
- Transfer functions
- Provenance records
- Derived products
- Citation information

The survey repository contains published products only.

It is not a processing environment.

It is not a time-series archive.

---

### gateway

The `gateway/` subdirectory is the submission service.

It receives uploaded survey packages and moves them through antivirus scanning, validation,
an engine preview build and curator review. Curator approval publishes the package as a git
commit to the survey repository.

Nothing entering the gateway is trusted until validation and review complete; a reviewed
git commit is its only output.

---

### engine

The `engine/` subdirectory is the offline processing engine.

Its role is to generate scientific products from survey packages.

Examples include:

- Apparent resistivity and phase curves
- Tipper and induction-arrow products
- Phase tensor products
- Dimensionality diagnostics
- Canonical EMTF XML and download bundles
- Decomposition products (planned)

Scientific processing occurs here rather than within the public portal.

This ensures:

- reproducibility
- version control
- consistent outputs
- independent validation

---

### portal

The `portal/` subdirectory is the public interface to AusMT.

Its responsibilities include:

- Search
- Discovery
- Visualisation
- Download services
- Citation support
- Machine-readable JSON products (a fixed, documented contract)

The portal consumes published products.

It does not perform scientific calculations.

This distinction is important.

A portal should remain lightweight, stable and easy to maintain.

Scientific processing belongs elsewhere.

---

### docs

The `docs/` subdirectory describes the system.

Its purpose is to document:

- Concepts
- Architecture
- Metadata models
- Scientific products
- Submission workflows
- Interoperability

The documentation is treated as a first-class component of the system.

---

## Information Flow

Information generally moves through AusMT in a single direction.

```text
Survey Package
↓
Validation
↓
Scientific Processing
↓
Publication
↓
Discovery
```

This flow reduces complexity and improves traceability.

Published products should not be modified by downstream systems.

---

## Trust Boundaries

The architecture contains several trust boundaries.

### Submission Boundary

External users may submit datasets.

Submitted material is not considered trusted until validation and review have been completed;
the gateway holds it in quarantine until then.

---

### Publication Boundary

Only validated and approved survey packages enter the published repository.

This repository represents the authoritative AusMT record.

---

### Processing Boundary

Derived products are generated by controlled workflows.

The public portal does not generate scientific products on demand.

This ensures that published outputs are reproducible and versioned.

---

## Collections, Surveys and Stations

AusMT organises data using three levels:

```text
Collection
↓
Survey
↓
Station
```

### Collection

A logical grouping of related surveys.

Examples:

- AusLAMP
- Institutional holdings
- State-based releases

---

### Survey

The primary scientific object within AusMT.

A survey package combines:

- Metadata
- Transfer functions
- Derived products
- Provenance
- Citation information

---

### Station

An individual observation location within a survey.

Stations contain the transfer-function products and metadata associated with a single site.

---

## Relationship to Time-Series Archives

AusMT does not store raw MT time-series data.

Long-term storage of time-series observations remains the responsibility of external repositories and archives.

Where available, survey packages record persistent identifiers linking transfer functions to the corresponding time-series collections.

This approach avoids duplication while maintaining traceability between observations and published products.

---

## Long-Term Architecture

The architecture is intentionally conservative.

Individual technologies will change over time.

Programming languages, frameworks, databases and storage systems will all evolve.

The core architectural concepts are expected to remain stable:

```text
Collection
↓
Survey
↓
Station

Metadata
Transfer Functions
Derived Products
Provenance
```

These concepts define the information model of AusMT and are intended to persist independently of any particular software implementation.
