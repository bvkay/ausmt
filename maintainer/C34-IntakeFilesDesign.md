# C34 — Intake generation of LICENSE.md and README.md (frozen design)

Trigger: olympic-dam-2004 published with a declared CC-BY-4.0 licence but no LICENSE.md or
README.md in the package, leaving permanent advisory WARNINGs. The declared licence id in
survey.yaml is authoritative and the engine already synthesises a deterministic LICENSE.txt
for download bundles from it; the archival package in the data repository should carry the
same rights statement and a human README from the moment of publication.

## D1. Generation point: the runner, at packaging time, before validation
When a submitted package lacks LICENSE.md or README.md, the gw-runner generates them into the
package after safe extraction and before the validator runs. The validator then passes its
structure checks in the same run; the curator reviews and approves the COMPLETE package
(generated files visible in the report bundle); the approve-what-you-publish property holds.
Rejected placements, recorded: client-side generation in add-survey (duplicates the rights
text in a second language; rights text must never drift) and publish-time injection (mutates
the package after curator approval).

## D2. Single source for the rights text
The engine's deterministic licence-text generation moves from build_portal into a new
STDLIB-ONLY leaf module `engine/extract/_license_text.py` (no numpy/mt_metadata imports),
imported by build_portal (bundle LICENSE.txt, byte-identical output — pinned by test) and by
the runner (LICENSE.md generation; the runner executes on the engine image where `extract`
is an installed package since C37). The gateway app itself never imports it.

## D3. Generation rules — fail closed
- LICENSE.md is generated ONLY for licence ids on the recognised allow-list the engine
  already maintains; an unrecognised id generates nothing and the validator WARNING stands.
- README.md skeleton from survey.yaml: name, organisation, year/dates, abstract, station
  count if cheaply known, citation guidance, licence line. No submitter contact details of
  any kind (the PII confinement invariant applies to generated content).
- Both files open with a machine stamp: "Generated by the AusMT gateway from the survey's
  declared metadata on <UTC date>. Replace via a metadata edit or resubmission."
- A package that ALREADY carries either file is untouched: existing bytes are never
  overwritten (byte-identity pinned by test).
- The manual-submission path (no gateway) is unchanged; its WARNINGs remain a correct signal.

## Tests (each able to fail)
- Package without the files: post-packaging both exist, carry the stamp, validator structure
  checks pass in the same run; LICENSE text equals the engine's bundle text for the same
  licence id (single-source proof).
- Package with either file present: byte-identical after packaging (no overwrite).
- Unrecognised licence id: no LICENSE.md generated; WARNING present.
- PII: generated files contain no submitter name/email/ORCID even when the DB record has them.
- build_portal bundle LICENSE.txt byte-identical before/after the D2 extraction (golden pin).
