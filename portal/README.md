# AusMT Portal

The AusMT Portal is a static discovery, visualisation, citation, and provenance interface for
Australian magnetotelluric (MT) transfer functions. It does not process MT data itself. All
catalogue, transfer-function, science, collection, and MTCAT products are generated by the
engine and served as static JSON documents.

AusMT is not a repository for raw MT time-series data. The portal provides discovery, visualisation,
provenance, and links to authoritative data holdings.

> **Maintaining or extending the portal?** See `docs` → `developer/architecture.md` for the
> system map, `developer/data-files.md` for the JSON the portal reads (positional arrays — read it
> before touching `src/data.js`/`main.js`), and `developer/extending.md` for how to add a filter,
> a catalogue column, or a plot.

---

## The AusMT ecosystem

AusMT is organised as the `ausmt` monorepo plus a separate data repository:

| Location | Purpose |
|------------|---------|
| portal/ (this directory) | Discovery, visualisation, citation, and contributor interfaces |
| engine/ | Transfer-function extraction, validation, science products, and catalogue generation |
| ausmt-surveys | Curated survey packages, metadata, transfer functions, and provenance |

Project-wide documentation, governance, standards, and MTCAT specifications live in `docs/` (the `maintainer/` knowledge base sits alongside).

---

## Features

- Interactive survey and station discovery
- Transfer-function visualisation
- Provenance and processing transparency
- Citation and export tools
- Collection and programme support (e.g. AusLAMP)
- Browser-based survey submission workflow (with on-map coordinate confirmation)
- MTCAT catalogue discovery document generation and exposure

## Running locally

The portal loads JSON products via browser `fetch`, so it must be served over HTTP (not `file://`):

```bash
python3 -m http.server 8000
# then open http://localhost:8000/
```

By default the portal reads its data from `./data/`. To consume data published elsewhere (e.g. the
engine's GitHub Pages), set `deployment.data_base_url` in `portal.config.yaml`.

---

## Configuration

Branding, deployment settings, and version are the single source of truth in `portal.config.yaml`.
After editing it, regenerate the browser-side reflection:

```bash
python3 tools/gen_config.py          # writes config.js
python3 tools/gen_config.py --check  # CI drift guard
```

## Data products

The portal expects generated products under `data/`:

```text
catalogue.json  tf.json  sci.json  surveys.json  collections.json  mtcat.json  build_provenance.json
```

These are generated by the engine. Their **positional** shape is defined in `docs` →
`developer/data-files.md` (and mirrored in the legend at the top of `src/data.js`).

---

## MTCAT

The portal exposes an MTCAT discovery document (`data/mtcat.json`) — an open JSON schema for
interoperable MT catalogue discovery, enabling exchange across independent MT portals while each
retains stewardship of its own holdings.

---

## Development

The portal is a static JavaScript application built from ES modules under `src/`. No server-side
processing is required. A headless smoke test and the Add-Survey logic test run via `pytest tests`
(see `.github/workflows/portal-ci.yml`). For architecture and contributor guidance, see `docs`.
